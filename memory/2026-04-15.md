# Memory — 2026-04-15

## QRIS Payment Flow — Bugs Found & Fixed

### Root Causes Identified

1. **PM2 `env_file` not supported in PM2 v6** — `.env` variables (like `WHATSAPP_SEND_QRIS_ON_CREATE`) were never loaded into the backend process. Fixed by manually parsing `.env` in `ecosystem.config.cjs` and spreading into `env: {}`.

2. **`whatsapp_sent_at` metadata dedup** — `sendQrisImageBestEffort` in `payments/service.js` checks `payment?.metadata?.whatsapp_sent_at` before sending. If evaluator already sent QR, subsequent calls from `/qris/direct` see this flag and skip. Customer may have received QR but API response says `whatsapp_sent: false`.

3. **Race condition: evaluator vs `/qris/direct`** — Evaluator auto-creates QRIS and sends WhatsApp before `/qris/direct` endpoint finishes. Response shows `whatsapp_sent: false` even though QR was actually delivered. Fixed API response to check payment metadata and return `sent_by: "evaluator"` when evaluator already sent.

4. **`sync-state.js` expected nested `orderContext` wrapper** — Agent writes state in flat format (no `orderContext` wrapper), so `sync-state.js` couldn't find `clientOrderId`, `paymentMethod`, etc. Fixed `sync-state.js` to handle both formats: `const ctx = state.orderContext || state;`

5. **Field name mismatch in flat state** — Flat format uses `fulfillment` (not `fulfillmentMethod`), `orderId` (not `clientOrderId`). Fixed `sync-state.js` to map: `ctx.fulfillmentMethod || state.fulfillment`, `ctx.clientOrderId || state.orderId`.

6. **Price mapping failure** — DB order items had `menu_name: "es kopi"` (from raw customer message) with no matching catalog entry. `calculateOrderAmount` threw "missing price mapping". Fixed `menuPricing.js` to fallback to item's own `price` field when catalog lookup fails.

7. **Agent not calling `sync-state.js`** — Agent says "QR sudah terkirim" without actually executing `node backend/sync-state.js sync <phone>`. This is an LLM behavior issue — agent writes state file but doesn't trigger backend sync. Manual sync works fine.

### WhatsApp CLI Switch

Backend switched from `openclaw message send` to `wacli` for sending WhatsApp messages. Key changes:
- `toOpenclawPhone` → `toWacliPhone` (formats as `<number>@s.whatsapp.net`)
- `runOpenclaw` → `runWacli` (calls `wacli send file --to <jid> --file <path> --caption <text>`)
- Binary: `WACLI_BIN` env var or default `wacli` (located at `/usr/local/bin/wacli`)

### Supabase Schema Notes

- Orders table: `orders` (columns: id, client_order_id, customer_phone_snapshot, customer_name_snapshot, status, etc.)
- Order items: `order_items` (columns: id, order_id, menu_id, menu_name, qty, unit_price, temperature, notes)
- Payments table: `order_payments` (NOT `payments`) — columns: id, order_id, provider_order_id, payment_status, amount, total_payment, qr_string, qr_image_url, expired_at, metadata (JSONB)
- Payment metadata includes: `whatsapp_sent_at`, `whatsapp_confirmed_sent`, `completed_at`
