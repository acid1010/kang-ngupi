## Date: 2026-04-16

## 🔴 CRITICAL Issues (Fix ASAP)

### 1. Supabase Client Created on Every Call — Memory Leak
**File:** `src/supabase.js`
**Issue:** `getSupabase()` creates a NEW Supabase client on every call (22 calls across codebase). Each client opens HTTP connections that may not be properly garbage collected.
**Impact:** Memory creep over time, especially in payment poller (called every 15s).
**Fix:** Singleton pattern — create client once, reuse.

### 2. .env File Permissions Too Open (664)
**File:** `backend/.env`
**Issue:** World-readable. Contains Pakasir API key, Supabase service role key, Pawoon credentials, JWT secret.
**Fix:** `chmod 600 backend/.env` and `chmod 600 backend/data/dashboard-users.json`

### 3. Password Hashing Uses SHA-256 (Weak)
**File:** `src/dashboard/auth.js`
**Issue:** `createHash('sha256').update(password + 'ngupi-salt')` — SHA-256 is fast, making brute-force easy. Static salt.
**Fix:** Use bcrypt or argon2 with per-user salt.

### 4. No Rate Limiting on Any Endpoint
**Issue:** No rate limiting on login, webhook, API endpoints. Vulnerable to brute-force login, webhook flooding, API abuse.
**Fix:** Add express-rate-limit to login (5 attempts/min), API (100 req/min), webhook (30 req/min).

### 5. CORS Too Permissive
**File:** `src/index.js` line 609
**Issue:** `cors({ origin: true })` allows ANY origin. SSE endpoint has `Access-Control-Allow-Origin: *`.
**Fix:** Restrict to `ngupingupi.me` and localhost for dev.

## 🟡 IMPORTANT Issues (Fix This Week)

### 6. No Log Rotation
**Issue:** `out.log` grows indefinitely (70KB now, will be MBs in production). No logrotate config.
**Fix:** Add PM2 log rotation module: `pm2 install pm2-logrotate`

### 7. No Backup Strategy for State Files
**Issue:** `state/orders-active/` files are local-only. Server crash = lost state.
**Fix:** State is also in Supabase (orders table), so this is mitigated. But add periodic git commit of state files.

### 8. Dashboard JWT Secret Weak
**Issue:** JWT secret is `ngupi-dashboard-<random>` — the prefix is predictable.
**Fix:** Use full random secret: `openssl rand -hex 32`

### 9. No Health Check Endpoint
**Issue:** No way to monitor if backend is alive from external monitoring.
**Fix:** Add `GET /health` that checks DB connection + returns uptime.

### 10. Payment Poller Memory Growth
**Issue:** Poller at 44MB after 6 hours. Supabase client recreation on every tick contributes.
**Fix:** Singleton Supabase client (see #1) + periodic GC hint.

### 11. No Request Timeout on Pawoon API Calls
**File:** `src/integrations/pawoon.js`
**Issue:** `fetch()` calls to Pawoon have no timeout. If Pawoon is slow/down, request hangs forever.
**Fix:** Add AbortController with 15s timeout.

## Architecture Assessment

### What's Solid ✅
- Dual payment verification (webhook + poller) — resilient
- Dedup protection at every level — no double sends
- Evaluator auto-creates QRIS — no manual trigger needed
- State machine in prompt — structured order flow
- Pawoon integration — POS sync automated
- Dashboard with SSE — realtime updates

### What's Fragile ⚠️
- WhatsApp connection (Baileys-based) — can break on protocol updates
- Single server — no redundancy
- 2GB RAM server — tight, swap already used (1.4GB swap)
- All services on one machine — backend, dashboard, OpenClaw, WhatsApp all compete for resources

### Recommended Architecture for Scale
1. **Separate WhatsApp gateway** from application server
2. **Redis** for dedup/caching instead of in-memory Maps
3. **Queue system** (BullMQ) for async processing instead of inline
4. **Separate DB** for orders (not Supabase free tier)
5. **CDN** for dashboard static assets
6. **Monitoring** (Uptime Robot + error alerting)
